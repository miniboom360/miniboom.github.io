How to Perform Effective Web3 Penetration Testing-Gathering Information (1)

September 30, 2023 in web3 penetration testing

In the web2 world, we often hear statements like this: “If you want to discover more vulnerabilities during penetration testing, collecting information is the first and most crucial step.” This is no exception in the penetration testing of the web3 world.

Information collection determines the scope of your testing. On the basis of considering breadth, it must also be refined; otherwise, you will be faced with a large number of assets and won’t know where to start.

However, it is unrealistic to achieve both breadth and refinement from the beginning. At least, it’s an irrational choice. Because doing this requires a significant amount of information collection and takes a long time, but it does not produce any vulnerabilities. Not only are we are easy to miss the low-hanging fruit, but the entire process will be boring.

Of course, you can automate the entire process of information collection, but that’s another topic we will discuss after covering the process and methodology of information collection.

My methodology is step-by-step, prioritizing breadth-first and then refining it in subsequent rounds. I strive to gain something from each round of information collection. If not, the next round will be more refined.

No more to talk, let’s just get into the formal part.

Find subdomains

There are many excellent subdomain collection tools on GitHub, but my favorite tool is Subfinder. Put the main domains into targets.txt, and you will obtain numerous subdomain results in subs_results.txt.

./subfinder -dL targets.txt -all -o subs_results.txt

I generally like to use other tools in addition to subfinder to create subdomains, such as oneforall.

After acquiring a large number of subdomains, you may encounter two questions.

What should I do next? Just check out those subdomains on ports 80 and 443 to find the website.

It’s a good idea, but some websites may be running on ports other than the default (e.g., 8443, 8080). Therefore, it would be a better choice to scan the ports of these subdomains next.

What is the scope of port scanning? Should I scan all ports from 1 to 65535?

Do you remember our principles? Step by step, it would be a better choice to scan the most common 1000 ports first. If you don’t gain anything this time, I recommend scanning all ports in the next round. :)

Port Fingerprinting

I recommend using the port scanner called kscan. The advantage of using kscan is that it integrates many excellent tools, not just port scanning, but also identifying website fingerprint information and grabbing the website’s title.

./kscan -t subs_results.txt --top 1000 -oC kscan_port_results.csv

Here, we can obtain an Excel file of kscan_port_results.csv which includes information such as port number, URL, protocol, and other details.

After this round, you can find most of the common port information, including HTTP, HTTPS, SSH, and some database ports like MySQL and Redis. Of course, you can also find that some websites are protected by WAFs, such as Cloudflare, etc.

Based on the ports information we have, there will be different strategies for processing. Such as database ports, SSH, FTP, etc., we can use brute force password methods. If it’s a high-risk port, such as port 445, we can use the ms0708 exploit to attack the target. But here, we give priority to protocols such as HTTP and HTTPS because they often deploy RESTful APIs and websites. It is easier for us to exploit vulnerabilities.

So, we can filter out HTTP and HTTPS contents in the “Service” column in the XLSX file, copy them out to prepare for the next web page fingerprinting, and put them in “http_subdomains.txt”.

Website Fingerprinting

Now that we already have many URLs in various forms such as http://subdomain.com, https://subdomain.com, and http://subdomain.com:8080, we may encounter these problems without any accidents.

For each URL, what is the title and status code? Based on Nginx or Tomcat? Which CMS should be used for development? Is there a WAF available?

Which URLs should we pay attention to? Which ones have higher priority?

Because of the existence of EHole, all the above questions have answers. All we need to do is run this command.

./ehole finger -l http_subdoamins.txt -o http_subdoamins_finger.xlsx

EHole will identify website fingerprints, WAF, and also provide a list of assets that need to be focused on. So, we should mark them and conduct a penetration test on them first.

Find more URLs

Because we have those URLs, such as http://subdomain/, http://subdomain:port/, and https://subdomain:port/, but it is not enough. We need to discover more URLs, which can be done through crawlers, brute force dictionaries, etc. Fortunately, there is a tool called “rad” that can help us crawl the URLs of a website and also discover APIs for us.

./rad --url-file http_subdomains.txt --text-output http_subdomains_urls.txt

Just need to wait a while, we will receive a large number of URLs in a text file. Some of the URLs may contain sensitive information, such as JSON files or important XLSX files. After this operation, we have obtained more URLs.

Website Screenshot

Remember that we have a large number of URLs, but we need to check them to determine which ones are login pages, which ones are management systems, and which ones are just static pages. This will help us prioritize which URLs to conduct a penetration test on first. Here, I use Gowitness to accomplish this.

./gowitness file -f ../findjs_url/http_subdomains_urls_without_http.txt

After running the above command, you can start an HTTP server by running the following command. It will display a visually appealing website page showing screenshots of all the URLs. Simply visit the URL http://127.0.0.1:7171.

./gowitness server

At this point, we can proceed with the initial round of penetration testing. We can set up a passive scanner to check the management website for weak passwords, SQL injection, or logic testing. What to do next completely depends on the individual.
